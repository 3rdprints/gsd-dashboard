# GSD Dashboard Distribution

## Source bundle

The v1.0 tag/release flow publishes a best-effort source bundle for offline-oriented rebuilds. Native installers remain the primary distribution channel, but the source archive should contain enough pinned inputs for a developer to audit and rebuild the release with minimal network access after the bundle is created.

Generate and validate the source bundle from the repository root:

```sh
scripts/release/build-source-bundle.sh --check
```

The bundle includes:

- `package.json` and `package-lock.json` for pinned Node dependency metadata.
- `Cargo.lock`, `src-tauri/Cargo.toml`, and `src-tauri/Cargo.lock` for pinned Rust dependency metadata.
- `vendor/` generated by `cargo vendor`.
- This `docs/distribution/BUILD.md` runbook.

Network caveat: creating the bundle still runs `npm ci`, `npm run build`, and `cargo vendor`, so the release machine needs access to npm and Cargo registries before the archive is produced.

## Cargo install fallback

`cargo install gsd-dashboard` is a developer fallback. The v1.0 tag/release flow is the first publish point for crates.io; do not use an early crates.io publish as a packaging test before native installers and updater artifacts are ready.

`cargo install gsd-dashboard` installs the binary without a native app bundle, auto-update support, or normal macOS Gatekeeper trust. Prefer the native installer for daily use.

Cargo packaging must include the built frontend assets from `src-tauri/dist/`, which is generated by `npm run build`. If `dist/index.html` is missing from the crate package during a release/package build, the build fails with:

```text
frontend assets missing: run npm run build before cargo package/install
```

## Release runbook

The release workflow signs updater artifacts with the Tauri updater key stored in GitHub Actions secrets. The private key must never be committed, pasted into issue text, or written into this runbook.

Required secret:

- `TAURI_SIGNING_PRIVATE_KEY`

Optional secret:

- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` only when the generated updater private key uses a password.

Before v1.0, confirm the updater private key has an offline backup outside the repository. Losing the updater private key permanently prevents existing installs from accepting future updates.
