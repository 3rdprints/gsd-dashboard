# Phase 01 Plan 03: Command Contracts Summary

**Managed Tauri AppState bootstrap with stable AppError/AppEvent contracts and thin boot/settings commands**

## Accomplishments

- Added `BootStatus` and expanded `AppState` to carry the SQLite pool, home path, app-data path, cache path, and startup readiness snapshot.
- Added a serializable `AppError` contract and tagged `AppEvent` enum for DB-as-truth invalidation.
- Moved runtime initialization into `bootstrap_app`, which resolves Tauri app-data paths, creates `cache.db`, runs migrations, initializes settings, and returns managed state.
- Added `get_boot_status`, `get_settings`, and `save_settings` commands backed by narrow generated Tauri permissions and command-level integration tests.

## Files Created/Modified

- `src-tauri/src/bootstrap.rs` - Runtime bootstrap for app-data cache creation, migrations, settings initialization, and AppState construction.
- `src-tauri/src/app_state.rs` - Managed state and serializable `BootStatus`.
- `src-tauri/src/error.rs` - Stable serializable `AppError` variants for command failures.
- `src-tauri/src/events.rs` - Tagged `AppEvent` enum with boot/settings invalidation variants.
- `src-tauri/src/commands/settings.rs` - Thin boot/settings commands and testable command helpers.
- `src-tauri/src/main.rs` - Tauri setup now manages bootstrapped AppState and registers the Phase 1 command handler list.
- `src-tauri/build.rs` and `src-tauri/permissions/autogenerated/*.toml` - Tauri app-command permission generation.
- `src-tauri/capabilities/default.json` - Allows only `core:default` plus the three Phase 1 commands.
- `src-tauri/tests/bootstrap.rs` and `src-tauri/tests/settings_commands.rs` - Integration coverage for contracts, boot readiness, command behavior, validation, and settings invalidation.

## Decisions Made

- Used Tauri's `app.handle().path().app_data_dir()` and `home_dir()` in production bootstrap; tests use `bootstrap_from_paths` to avoid writing into the real user app-data directory.
- Kept commands thin by delegating reads/writes to `settings::load_or_initialize` and `settings::save`.
- Used generated `allow-*` app permissions rather than broad capability entries or filesystem permissions.
